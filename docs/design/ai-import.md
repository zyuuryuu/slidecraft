# AI-Import — 乱雑なテンプレを正確に「理解」して取り込む

> 位置づけ: 「Re-make（変換）」ではなく **AI-Import（理解を助ける取り込み）**。AI はテンプレを
> 変換するのではなく、**乱雑なテンプレの幾何を正確に理解**する（ロール/構造の同定）ことを助ける。
> 関連: [ADR-0027](../adr/0027-remake-source-visual-preservation.md)（faithful=視覚保持は土台）、
> [ADR-0025](../adr/0025-placeholder-role-resolution.md)（gate 付き title リカバリ）、
> [ADR-0023](../adr/0023-third-party-master-idx-convention.md)（第三者マスターのロール曖昧）。

## 1. なぜ AI-Import か

- 実在の会社テンプレは**乱雑**：idx 任意、body 型なのに実はタイトル、装飾シェイプ/メタ（頁番号/
  日付/ロゴ/罫線）と本文枠の混在、無意味な命名。
- 目標は「**どんなテンプレでも使える**」（guardrail_any_template）＝**乱雑なテンプレを正確に理解
  できるか**に尽きる。理解できれば Import も faithful Re-make も束縛もプレビューも成立する。
- 決定論ヒューリスティック（`placeholderRole`/`classifyLayout`）は綺麗なテンプレには強いが乱雑さ
  には**天井**（ADR-0023 の残エッジ）。
- AI の価値＝**人間が見れば分かる曖昧判断**（「上端・幅広・大フォント → 型が body でもタイトル」）。
  変換ではなく**理解のラストマイル**。

**「Make か？」への答え**: いいえ。パース＝**理解（comprehension）**で、新しい幾何を著述しない。
Import（理解して使う）→ faithful（理解して保存）→ canonical Re-make（理解して標準へ再構築）→
Make（ゼロから著述）は一本のスペクトラムで、共通項は全部「**正確な理解**」。投資先はここ。

## 2. 課題（＝理解の難所）

1. **ロール同定**：どれがタイトル/本文/サブか。型・idx が当てにならず幾何が主 signal だが曖昧。
2. **content / decoration / meta の三分**：本文枠か・装飾か・頁番号/日付/ロゴ/罫線か。見た目が似る。
3. **構造判定＋グルーピング**：本文領域は何個か、2カラム/3カード。小枠の束ね（箱＋ラベル＋アイコン＝1ユニット）。
4. **レイアウトの目的**：表紙/章扉/本文/比較/締め。名前が無意味なとき幾何＋テキストから推定。
5. **継承・group 変換の解決**：master/layout 継承、group 入れ子 xfrm を正確に。
6. **束縛写像**：理解結果を原稿(Markdown)の内容へどう割り当てるか。

**メタ課題**：AI は不安定・高コスト → 使いすぎない・検証する・人間が最終確認する枠組みが必須。

## 3. ハーネス設計（harness-over-model・5段）

決定論で殆どを固め、AI は曖昧判断だけ、全部を検証、人間が最終確認：

1. **決定論 DISTILL（機械的・確実）**：幾何を正確にパース（継承・group 解決）、型・テキスト・idx を
   **AI が噛める構造化サマリへ蒸留**（生 XML は渡さない）。
2. **決定論 高信頼推論**：明白なケース（型付き sldNum/dt/ftr＝メタ、上端幅広＝タイトル、下記の幾何
   メタ検出）は AI 無しで確定。**各判断に信頼度を付け、曖昧だけをフラグ**。
3. **AI ラストマイル判断**：曖昧フラグ付きの判断**だけ**を蒸留サマリと共に AI へ。出力は**構造化＋理由**。
4. **検証（validate-and-retry・ADR-0018）**：解釈が**束縛可能で幾何的に整合するか**を検証 → 不整合
   は retry、駄目なら**決定論へ never-worse フォールバック**。
5. **人間 in the loop**：理解結果（プレビュー＋ロールラベル）を取り込み結果バーで提示し確認・修正。

**原則**：AI は判断に・機械は機構に／入力を蒸留（小モデルでも効く）／出力を検証／信頼度ゲート／人間が最終審判。

## 4. 実データ棚卸し（2026-07-08・決定論パースの現状精度）

`配布資料_公文書高密度`／`技術報告`／`ビジュアルデッキ` の 39 レイアウトを監査 → **17 で同定のズレ**。
分類：

- **A. メタ/装飾の body 誤認（最頻・最重要）**：`body@30(0.5,6.7 12.3x0.3)`＝最下部の全幅極薄ライン
  （フッター罫線）、`body@31(9.7,0.3 3.1x0.6)`＝右上小片（見出しタグ/頁要素）が**全レイアウトで body
  に誤分類**。master 継承のデザイン要素でコンテンツ枠ではない。技術報告の出典/脚注行（y=6.3）も同様。
  → 根因：**幾何ベースのメタ検出が無い**（フッター帯の極薄／隅の小片を位置・サイズで見抜けていない）。
- **B. bodyCount 過大**：A の偽 body ＋ ブロック見出し（`title@15`/`subtitle@16`）を別カウント →
  「本文3ブロック」が bc=6（実 3）。列比も @30(幅12.3) vs @31(幅3.1) で歪む。
- **C. 主 body/構造の真の曖昧（＝AI 向き）**：KPI/カード3列/プロセスの多数小枠の**束ね方**は幾何だけ
  では曖昧。

**含意**：A/B は**決定論の幾何ルールで直せる**（ハーネスの仕事）。C だけ AI（ラストマイル）。まず
A/B を潰すと flag が大幅に減り、残る真の曖昧＝C に AI を集中できる。

## 5. いい仕事にする鍵＝計測

「良い」を測れないと育たない。**ゴールデン集（実テンプレ＋人手の正解ロール/構造）**で決定論パーサの
**同定精度をベースライン計測** → 各改善（幾何メタ検出→AI-assist）の**差分を計測**。これが harness を
育てるループ。監査スクリプトはその骨組み。
