# ADR-0031: 保守性ゲート — 構造規約の実行可能化と「一致テスト」規則

- Status: Accepted
- Date: 2026-07-18

## Context

BindingPlan 事件（ADR-0030）: 束縛写像が 7 箇所で独立再計算され、#124/#135/#144 の実バグを生んだ。
再発防止策を検討する過程で実測した事実:

1. **従来型メトリクスはこの欠陥を検出できない。** 行数・複雑度・コピペ率はいずれもファイル単体の
   指標であり、欠陥は「同じ意味を複数箇所で再計算する」というファイル**間**の関係に宿っていた。
2. **既存の規約は散文で、実行可能でない。** R1（400 行）は明文規則だが 2026-07-18 時点で 15 ファイル
   が超過しており、eslint には max-lines も import 制限も未設定。R2（engine 純度）もレビュー頼み。
3. **文字列 grep の規約検査は偽陽性で使い物にならない**（html-shell の埋め込み viewer JS 内
   `document.`、OOXML の `officedocument` 文字列で実測）。検査は import 文の解析ベースが必要。

## Decision

3 層のゲートを導入する。導入基準は「BindingPlan 型の欠陥（権威の迂回・意味の重複・silent
fallthrough）を捕まえるか」であり、汎用メトリクスの網羅ではない。

### G1: 構造規約テスト（CI 必須・fail する）

`tests/arch-conformance.test.ts`。vitest 内なので既存 CI の必須 `test` ジョブに自動で乗る
（新ツール依存ゼロ・検査は import 文解析ベース）:

- **権威ルール**: `bindContentByRole`/`expandGroups` の import 元は許可リストのみ
  （ADR-0030 の再発防止本体。新しい再計算サイトの追加で赤）
- **R2 純度**: engine → components/ipc/mcp/react/@tauri-apps 等の import 禁止
- **R1 行数**: 400 行。既存超過 15 件は凍結リスト（新規超過で赤・棚卸しテスト付き）
- **循環 import 禁止**（engine 内・実行時エッジのみ。type-only は無害なので除外。
  既知の意図的循環 schema⇄schema-diagnostics は凍結）

**ratchet 運用**: 凍結/許可リストは「現状の凍結」。増やす変更は原則差し戻し（増やすなら PR で
理由明示）。分割・統合が進んだらリストから消す — **リストの縮小が保守性改善の実測値**。
ADR-0030 段階 C–E の完了は権威許可リストの縮小として現れる。

### G2: 「一致テスト（agreement test）」規則（プロセス・CLAUDE.md R8）

意味の重複は静的解析で測れないため、規則として運用する:

> 同じ意味（写像・判定・容量計算）を 2 経路で計算する場合、(a) 一本化する、または
> (b) 両経路の**一致テスト**を置く。どちらも無い PR は差し戻す。

これは ADR-0030 段階A の「`assignments == bindContentByRole`」・段階B の「新旧経路
byte-identical」ゲートの型への昇格であり、新発明ではない。PR テンプレのチェック項目にも載せる。

### G3: 傾向観測センサス（fail しない・on-demand）

コピペ率（jscpd）と hotspot（git churn × 行数）を census スクリプトとして整備し、定期的に眺めて
次の分割候補（#129 型）を決める材料にする。**ゲートにはしない**（閾値ゲート化は Goodhart の法則で
数字合わせを誘発する）。実装は別 issue。

### やらないこと

- カバレッジ % 目標・SonarQube 類の常設ダッシュボード・複雑度閾値の fail ゲート。
  BindingPlan 型の欠陥を捕まえない割に偽陽性が多く、G1/G2 と責務が重複する。

## Consequences

- (+) R1/R2/権威/循環が「レビューで気づく」から「CI で赤くなる」へ。散文規約の実行可能化。
- (+) 凍結リストが分割バックログの単一の源になる（#129 の後継が一覧化された状態）。
- (+) G2 により、経路の複製が入る PR はレビュー基準が明文化される。
- (−) 正当な構造変更（権威の移動等）ではリスト更新が必要 = PR に 1 手増える。それが「議論の
  トリガー」として機能するのが狙いなので、コストではなく仕様。
- (−) import 解析は正規表現ベースの近似（エッジ漏れ側に保守的）。厳密化が必要になったら
  dependency-cruiser 等への置換を検討（現時点では依存追加に値しない）。
